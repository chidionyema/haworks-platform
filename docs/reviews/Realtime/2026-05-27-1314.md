## CRITICAL: Message Loss in SignalR Hub Connection

### CRITICAL: SignalR message delivery failure not handled
- **File**: src/Realtime/Realtime.Api/Infrastructure/SignalR/NotificationHub.cs:55-62
- **Issue**: If any `SendAsync` call fails during the foreach loop, the method throws an exception, but `AcknowledgeMessagesAsync` still executes in the finally block (implicitly), causing message loss
- **Impact**: Users will permanently lose notifications if SignalR delivery fails, violating at-least-once delivery guarantee
- **Fix**: Wrap SendAsync in try-catch, track successful deliveries, only acknowledge successfully sent messages:
```csharp
var successfulCount = 0;
foreach (var msg in messages)
{
    try 
    {
        await Clients.Caller.SendAsync("ReceiveNotification", msg);
        successfulCount++;
    }
    catch (Exception ex)
    {
        _logger.LogError(ex, "Failed to send message {MessageId} to user {UserId}", msg.MessageId, userId);
        break; // Stop sending to avoid partial state
    }
}
await _inboxService.AcknowledgeMessagesAsync(userId, successfulCount);
```

### MEDIUM: Inbox size limit not enforced atomically
- **File**: src/Realtime/Realtime.Api/Infrastructure/Persistence/RedisInboxService.cs:47
- **Issue**: `LTRIM` to `MaxInboxSize-1` happens after `LPUSH`, creating a brief window where inbox exceeds limit
- **Impact**: Redis memory could spike if many messages arrive rapidly before trim executes
- **Fix**: Use Lua script for atomic LPUSH+LTRIM:
```csharp
const string script = @"
    redis.call('LPUSH', KEYS[1], ARGV[1])
    redis.call('LTRIM', KEYS[1], 0, tonumber(ARGV[2]))
    redis.call('EXPIRE', KEYS[1], tonumber(ARGV[3]))
    return 1";
await db.ScriptEvaluateAsync(script, new RedisKey[] { inboxKey }, 
    new RedisValue[] { json, InboxConstants.MaxInboxSize - 1, (int)TimeSpan.FromDays(InboxTtlDays).TotalSeconds });
```
