# Principal Engineer Code Review: Merchant Service

## CRITICAL: Race Condition in Merchant Creation
- **File**: src/Merchant/Merchant.Application/Merchants/Commands/CreateMerchant/CreateMerchantCommand.cs:29-32
- **Issue**: Classic check-then-act race condition. Two concurrent requests can both pass the `AnyAsync` checks for existing owner/slug and then both attempt to insert, causing database constraint violations or duplicate merchants.
- **Impact**: Data integrity violations, duplicate merchant creation, or runtime exceptions in production under concurrent load.
- **Fix**: Use pessimistic locking with raw SQL or implement a unique constraint + idempotency key pattern:
```csharp
// Option 1: Optimistic approach with proper exception handling
try {
    var merchant = MerchantProfile.Create(request.OwnerId, request.Name, request.Slug);
    _context.Merchants.Add(merchant);
    await _context.SaveChangesAsync(cancellationToken);
} catch (DbUpdateException ex) when (IsUniqueConstraintViolation(ex)) {
    // Check which constraint was violated and return appropriate error
    var existing = await _context.Merchants.FirstOrDefaultAsync(m => m.OwnerId == request.OwnerId || m.Slug == request.Slug);
    throw new InvalidOperationException(existing?.OwnerId == request.OwnerId ? "Owner already has merchant" : "Slug already exists");
}
```

## CRITICAL: Event-Transaction Ordering Bug
- **File**: src/Merchant/Merchant.Application/Merchants/Commands/ApproveMerchant/ApproveMerchantCommand.cs:25-32
- **Issue**: Events are published to outbox BEFORE domain state changes are saved. If SaveChangesAsync fails, the event is in the outbox but the state change is rolled back, leading to inconsistent state.
- **Impact**: Event consumers will receive MerchantActivatedEvent but the merchant remains in Pending status, causing downstream system inconsistencies.
- **Fix**: Reorder operations or use a transaction scope:
```csharp
merchant.Activate(request.ApprovedBy);
await _context.SaveChangesAsync(cancellationToken); // Save first
await _publishEndpoint.Publish(new MerchantActivatedEvent { MerchantId = merchant.Id }, cancellationToken);
```

### HIGH: Domain State Machine Validation Gaps
- **File**: src/Merchant/Merchant.Domain/Aggregates/MerchantProfile.cs:29-35
- **Issue**: Activate method allows transition from Suspended to Active, but domain logic doesn't verify the suspension reason was resolved or who is authorizing the reactivation.
- **Impact**: Business rule violations, merchants reactivated without proper approval process.
- **Fix**: Add validation for reactivation from suspension:
```csharp
public void Activate(string approvedBy)
{
    if (Status == MerchantStatus.Suspended && string.IsNullOrEmpty(approvedBy))
        throw new InvalidOperationException("Cannot reactivate suspended merchant without explicit approval");
    
    if (Status is not (MerchantStatus.Pending or MerchantStatus.Suspended))
        throw new InvalidOperationException($"Cannot transition from {Status} to Active");
    
    // Clear suspension data when reactivating
    if (Status == MerchantStatus.Suspended)
    {
        SuspensionReason = null;
        SuspendedAt = null;
        SuspendedBy = null;
    }
    
    Status = MerchantStatus.Active;
    ApprovedAt = DateTimeOffset.UtcNow;
    ApprovedBy = approvedBy;
}
```
