## CRITICAL: Concurrency Race Condition in Tax Rate Updates

### CRITICAL: Check-Then-Act Race in AdminTaxRatesController
- **File**: src/Pricing/Pricing.Api/Controllers/Admin/AdminTaxRatesController.cs:60-75
- **Issue**: Tax rate update uses check-then-act pattern without proper locking. Between `GetByIdAsync` and `SetEffectiveTo`, another request could modify the same tax rate.
- **Impact**: Could create overlapping tax rate periods or lost updates, leading to incorrect tax calculations and compliance issues.
- **Fix**: Use pessimistic locking with FOR UPDATE or implement optimistic concurrency with row versioning.

## HIGH: HTTP Client Error Handling Flaw

### HIGH: Overly broad exception categorization
- **File**: src/Pricing/Pricing.Infrastructure/Http/CatalogPricingHttpClient.cs:42-47
- **Issue**: All `HttpRequestException` instances are treated as transient failures and rethrown for Polly retry, but 4xx errors (404, 401, 403) are permanent and shouldn't be retried.
- **Impact**: Unnecessary retry storms on permanent failures, increased latency, potential catalog service overload.
- **Fix**: Distinguish permanent vs transient errors:
```csharp
catch (HttpRequestException ex) when (ex.Data.Contains("StatusCode"))
{
    var statusCode = (HttpStatusCode)ex.Data["StatusCode"];
    if (statusCode >= HttpStatusCode.BadRequest && statusCode < HttpStatusCode.InternalServerError)
    {
        _logger.LogWarning("Permanent failure fetching product {ProductId}: {StatusCode}", id, statusCode);
        return new CatalogProductResponse { IsSuccess = false };
    }
    // Only retry 5xx errors
    throw;
}
```

### HIGH: Dual responsibility pattern creates consistency risk
- **File**: src/Pricing/Pricing.Api/Controllers/PricingController.cs:70-74
- **Issue**: Handler doesn't save changes (for consumer compatibility), but controller must call `SaveChangesAsync` manually. If forgotten, audit logs are lost.
- **Impact**: Missing audit trail for price calculations, compliance violations.
- **Fix**: Create separate handlers for HTTP vs messaging scenarios, or use a post-processing filter.
