## CRITICAL: Coordinate Logic Bug in CreateAddressCommand

### CRITICAL: Inconsistent coordinate handling allows invalid geocoding fallback
- **File**: src/Location/Location.Application/Commands/CreateAddressCommand.cs:32-34
- **Issue**: Default initialization `double lat = request.Latitude ?? 0; double lon = request.Longitude ?? 0;` combined with partial coordinate checking creates a critical bug. If latitude=50 is provided but longitude is null, the code sets lon=0, attempts geocoding (due to null check), but if geocoding fails, it will use the invalid coordinates (50, 0) instead of rejecting the request.
- **Impact**: Invalid addresses could be stored in production with coordinates pointing to incorrect locations (e.g., off the coast of West Africa at 0° longitude).
- **Fix**: 
```csharp
// Remove default initialization
double lat, lon;

if (!request.Latitude.HasValue || !request.Longitude.HasValue)
{
    // Ensure both are null if either is null
    if (request.Latitude.HasValue || request.Longitude.HasValue)
    {
        return Result.Failure<Guid>(Error.Validation("Coordinates.Partial", "Both latitude and longitude must be provided together."));
    }
    // Continue with geocoding...
}
else
{
    lat = request.Latitude.Value;
    lon = request.Longitude.Value;
}
```
