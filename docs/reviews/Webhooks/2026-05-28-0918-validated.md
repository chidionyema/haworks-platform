## CRITICAL: Plaintext Secret Storage Security Vulnerability

### CRITICAL: Storing webhook secrets in plaintext alongside hashes
- **File**: src/Webhooks/Webhooks.Domain/WebhookSubscription.cs:9
- **Issue**: The domain entity stores both `Secret` (plaintext) and `SecretHash` (bcrypt) properties. The plaintext secret is persisted to the database and accessible to anyone with DB access.
- **Impact**: Complete compromise of webhook security. Attackers with database access can read all partner webhook secrets, enabling them to forge webhook signatures and impersonate the platform.
- **Confidence**: 10/10
- **Fix**: Remove the `Secret` property entirely. Only store the hash. For HMAC signing, retrieve the secret from a secure vault (Azure Key Vault, HashiCorp Vault) using the subscription ID as the key reference.

### CRITICAL: Authorization bypass via empty GUID
- **File**: src/Webhooks/Webhooks.Api/Controllers/SubscriptionsController.cs:20
- **Issue**: `GetPartnerId()` returns `Guid.Empty` when claim parsing fails, but operations proceed with this empty GUID
- **Impact**: Users could potentially access or modify subscriptions that don't belong to them if database queries match empty GUIDs
- **Confidence**: 10/10
- **Fix**: Throw `UnauthorizedAccessException` when partner ID claim is missing or invalid instead of returning `Guid.Empty`

```csharp
private Guid GetPartnerId()
{
    var raw = User.FindFirst("partner_id")?.Value ?? User.FindFirst(ClaimTypes.NameIdentifier)?.Value;
    if (!Guid.TryParse(raw, out var id) || id == Guid.Empty)
        throw new UnauthorizedAccessException("Valid partner ID claim is required");
    return id;
}
```
