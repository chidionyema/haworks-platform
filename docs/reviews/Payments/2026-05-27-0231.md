## CRITICAL: Idempotency Keys Defeat Their Own Purpose

### CRITICAL: Broken Idempotency Pattern Across Controllers
- **File**: src/Payments/Payments.Api/Controllers/Admin/StripeKeyRotationController.cs:32
- **Issue**: `IdempotencyKey = Guid.NewGuid().ToString("N")` generates a new GUID on every request, defeating idempotency
- **Impact**: Duplicate requests will create multiple Stripe key rotations instead of being deduplicated
- **Fix**: Accept idempotency key from client or derive from request content hash
```csharp
IdempotencyKey = request.IdempotencyKey ?? 
    Convert.ToBase64String(SHA256.HashData(Encoding.UTF8.GetBytes($"{request.NewSecretKey}:{DateTime.UtcNow:yyyyMMdd}")))
```

### CRITICAL: Same Idempotency Issue in Multiple Controllers
- **File**: src/Payments/Payments.Api/Controllers/RefundsController.cs:26
- **File**: src/Payments/Payments.Api/Controllers/SubscriptionsController.cs:42, 72, 90
- **Issue**: All use `Guid.NewGuid().ToString("N")` for idempotency keys
- **Impact**: Payment operations are not idempotent, risking duplicate charges/refunds
- **Fix**: Accept client-provided idempotency keys or derive deterministically

## CRITICAL: Authorization Bypass Vulnerabilities

### CRITICAL: Missing Payment Ownership Validation
- **File**: src/Payments/Payments.Api/Controllers/RefundsController.cs:18-35
- **Issue**: No verification that the authenticated user owns the payment being refunded
- **Impact**: User A could create refunds for User B's payments (IDOR vulnerability)

```csharp

if (payment?.UserId != HttpContext.GetForwardedUserId())
    return Forbid();
```
