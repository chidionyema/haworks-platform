## CRITICAL: SaveChanges Pattern Violation in Controller

### CRITICAL: Controller breaks dual-use handler pattern
- **File**: src/Notifications/Notifications.Api/Controllers/NotificationsController.cs:37
- **Issue**: Controller calls `_repository.SaveChangesAsync()` directly after the handler returns, but `SendNotificationCommandHandler` is documented as "dual-use" and explicitly states "Do NOT call SaveChangesAsync here" (lines 124-126)
- **Impact**: Breaks transaction boundaries for MassTransit consumers. When called from RefundEmailConsumer, this creates nested transaction issues and bypasses the outbox pattern
- **Fix**: Remove SaveChanges call from controller. Add SaveChanges to HTTP-only code path or make handler handle its own transaction boundaries
- **Confidence**: 10

## CRITICAL: Broken Idempotency in Channel Gateway

### CRITICAL: Random GUID breaks provider idempotency
- **File**: src/Notifications/Notifications.Infrastructure/Channels/Email/EmailChannelGateway.cs:115
- **Issue**: `var providerMessageId = result.ProviderMessageId ?? Guid.NewGuid().ToString("N");` generates random GUID when provider doesn't return message ID
- **Impact**: Same notification retry gets different provider message IDs, breaking webhook correlation and delivery tracking
- **Fix**: Use deterministic ID based on notification ID + provider name: `result.ProviderMessageId ?? $"{notification.Id:N}-{provider.Name}"`
