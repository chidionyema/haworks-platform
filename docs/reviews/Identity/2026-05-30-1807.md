## COMPREHENSIVE CODE REVIEW: Identity Service

### CRITICAL: Missing Refresh Token Cleanup Strategy
- **File**: src/Identity/Identity.Infrastructure/RefreshTokenService.cs
- **Issue**: No background service or scheduled job to clean up expired refresh tokens from the database
- **Impact**: Database will accumulate expired refresh tokens indefinitely, causing storage bloat and potential performance degradation
- **Fix**: Implement a background hosted service to periodically delete expired refresh tokens: `DELETE FROM RefreshTokens WHERE Expires < NOW()`

### CRITICAL: Revoked Token Storage Without TTL Cleanup  
- **File**: src/Identity/Infrastructure/TokenRevocationService.cs
- **Issue**: Revoked tokens are stored in database without automatic cleanup after their natural expiry
- **Impact**: Database will accumulate revoked token records indefinitely, even after the original tokens would have expired naturally
- **Fix**: Add background cleanup job: `DELETE FROM RevokedTokens WHERE CanBeCleanedUp = true`

### HIGH: Potential Username Race Condition in External Login
- **File**: src/Identity/Application/ExternalLoginCallbackCommandHandler.cs:198-210
- **Issue**: `GetUniqueUserNameAsync` checks for username availability, but another concurrent registration could claim the same username before this transaction commits
- **Impact**: Could result in duplicate username errors or inconsistent state during high concurrent external login attempts
- **Fix**: Wrap the username check and user creation in a single database transaction with appropriate isolation level, or use a unique constraint with retry logic

### HIGH: Missing Rate Limiting on Critical Endpoints
- **File**: src/Identity/Api/Controllers/AuthenticationController.cs:96,130,168
- **Issue**: Verify-token, logout, and service-token endpoints lack rate limiting protection
- **Impact**: These endpoints could be abused for denial of service or brute force attacks
- **Fix**: Apply `[EnableRateLimiting("auth")]` to verify-token, logout, and implement separate rate limiting policy for service-token endpoint

### HIGH: JWT Generation Performance - N+1 Query Pattern
- **File**: src/Identity/Infrastructure/JwtTokenService.cs:63-68
- **Issue**: Generates JWT by making separate queries for roles (`GetRolesAsync`) and claims (`GetClaimsAsync`) for each user
- **Impact**: Two separate database queries per JWT generation reduces performance under load
- **Fix**: Implement a single query that retrieves user, roles, and claims in one operation, or add caching for user roles/claims

### MEDIUM: External Login Email Verification Trust Issue
- **File**: src/Identity/Application/ExternalLoginCallbackCommandHandler.cs:188-198
- **Issue**: `IsEmailVerified` method trusts "Google", "Microsoft", "Apple" providers without checking the `email_verified` claim
- **Impact**: If these providers change their behavior or if tokens are compromised, unverified emails could be accepted
- **Fix**: Always check for explicit `email_verified` claim first, use provider trust as fallback only
